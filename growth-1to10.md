# 앱 수익 처방전 — 솔루션 개발 계획서

> 버전: v1.3 | 작성일: 2026.02.28 | 수정일: 2026.02.28
> 목적: "앱 수익 처방전" 제품의 데모 영상 제작 → 사전 신청 확보 → 프리토타입 개발

---

## 개요

제품을 먼저 만들고 고객을 찾는 게 아니라, **데모 영상으로 고객을 먼저 확보한 뒤 제품을 만든다.**

```
[Phase 1] 더미 데이터로 데모 영상 제작
    ↓
[Phase 2] 영상 홍보 → 사전 신청 수집
    ↓
[Phase 3] 검증된 수요 기반으로 프리토타입 제작
```

데모 영상이 보여줄 것: **"앱 데이터를 넣으면 이런 처방전이 나옵니다"** — 실제 작동하지 않아도 된다. 결과물의 가치가 느껴지면 된다.

### 제품 형태

**개발 환경에 녹아드는 수익화 어시스턴트.**

사용자는 이미 Claude Code, Cursor, Codex 같은 AI 코딩 도구 안에서 일한다. 이 솔루션은 **한 번 설치하면, 이후 결제 기간 동안 별도 설정 없이 개발자가 코딩하는 동안 자동으로 함께한다.**

```
[사용자 경험]

1. 설치 (1회, 30초)
   $ npx @apprx/cli init
   → 프로젝트에 .apprx.md 생성 + AI 도구에 자동 연결

2. 이후 — 그냥 코딩하면 됨
   사용자: "구독 결제 구현해줘"
   AI: (이미 이 앱이 Level 2이고, ₩2,900 구독이 추천되고,
        RevenueCat이 적합하다는 걸 안다)
       → 맥락에 맞는 코드를 바로 생성

3. 확인이 필요할 때
   사용자: "내 앱 수익화 현황 보여줘"
   AI: 진단 결과, 벤치마크, 이번 주 할 일을 즉시 답변
   또는: 웹 대시보드에서 시각적으로 탐색
```

**두 가지 인터페이스:**

| 인터페이스 | 역할 | 특징 |
|---|---|---|
| .apprx.md (컨텍스트 파일) | AI 어시스턴트에게 수익화 맥락 제공 | 원커맨드 설치, 제로 설정, 어디서든 작동 |
| 웹 대시보드 | 시각적 탐색, 진단 과정/근거 확인 | 차트, 비교 테이블, 인터랙티브 UI |

둘은 같은 데이터를 보여주되, .apprx.md는 개발자의 **일상 워크플로우**에, 웹 대시보드는 **깊은 확인**에 쓰인다.

### 가격 모델

| 플랜 | 가격 | 포함 내용 | 근거 |
|---|---|---|---|
| 2주 사용권 | ₩4,900 | 대시보드 접근 + 진단서 + 2주간 실행 가이드 | 토스 CC(Conversion Cycle) 기준 — 지표 세팅~첫 데이터 수집까지 최소 2주 |
| 월간 구독 | ₩49,000/월 | 대시보드 지속 접근 + 주간 코칭 + 전략 업데이트 | 연속 실험 사이클 지원 |

### 사용자 여정 분기

고객의 **측정 준비도**에 따라 대시보드가 다른 경로를 제공한다:

```
[온보딩] 측정 준비도 체크
    │
    ├─ Case A: 미설정 (GA/AdMob/결제API 없음)
    │   → 1차 가치: 지표 측정 방법 안내 (뭘 달아야 하는지)
    │   → 2차 가치: 2주간 측정 실행 방안 (어떻게 읽는지)
    │   → 2주 후: 수집된 데이터로 본 진단 실행
    │
    ├─ Case B: 부분 설정 (일부만 있음)
    │   → 보완 포인트 안내 + 기존 데이터로 부분 진단
    │   → 보완 완료 후 전체 진단 업데이트
    │
    └─ Case C: 완전 설정 (GA + 수익 추적 완비)
        → 바로 그로스 전략 제안
        → 대시보드 즉시 활성화
```

이 분기가 **2주 사용권의 존재 이유**다. Case A 고객은 "지금 당장 진단서를 받는 것"이 아니라 "2주 동안 측정 체계를 갖추는 것" 자체가 첫 번째 가치다.

### 핵심 자산

이를 위해 3개의 핵심 자산을 설계한다 (데모용 → 실제 제품용으로 점진 발전):

| # | 자산 | 역할 | 비유 |
|---|---|---|---|
| 1 | 수익화 분석 프롬프트 시스템 | 두뇌 | 의사의 진단 역량 |
| 2 | 벤치마크 데이터 시트 | 근거 | 의사의 검사 결과 참조표 |
| 3 | 웹 대시보드 | 시각적 산출물 | 처방전 열람실 |
| 4 | .apprx.md (컨텍스트 파일) | 개발 환경 통합 | 의사가 항상 옆에 있는 것 |

이 4개가 조합되면:
```
[고객 앱 데이터] + [벤치마크 참조] → [프롬프트 시스템] → [웹 대시보드 + .apprx.md]
                                                            ↑ 시각적 탐색
                                                            ↑ AI 어시스턴트가 자동 참조
```

**Phase 1(데모)에서는** 더미 고객 데이터로 웹 대시보드 + .apprx.md 샘플을 만들고, "설치부터 AI가 수익화 맥락을 이해하기까지" 과정을 영상으로 녹화한다.

## 1. 수익화 분석 프롬프트 시스템

### 1-1. 목적

Claude 프로젝트에 내장할 시스템 프롬프트. 고객의 앱 데이터를 입력하면, 구조화된 분석과 처방을 자동으로 생성한다. 승원은 결과물을 검수하고 보완하는 역할.

### 1-2. 시스템 구조

프롬프트는 6개의 모듈로 구성된다. Module 0이 게이트 역할을 하며, 측정 준비도에 따라 이후 경로가 달라진다.

```
Module 0: 측정 준비도 진단 (Gate)
    │
    ├─ 미설정 → 측정 세팅 가이드 + 2주 실행 방안 (여기서 일단 끝)
    ├─ 부분 설정 → 보완 안내 + 가능한 범위에서 Module A~E 실행
    └─ 완전 설정 → 풀 진단
          │
          ↓
    Module A: 앱 프로필 파싱 & 정규화
        ↓
    Module B: 수익화 단계 진단 (Level 1~5)
        ↓
    Module C: 벤치마크 비교 분석
        ↓
    Module D: 수익 시뮬레이션 & 최적 모델 매칭
        ↓
    Module E: 실행 로드맵 생성
```

### 1-3. 모듈별 상세 설계

#### Module 0: 측정 준비도 진단

**Input:** 고객이 온보딩 폼에서 선택한 현재 세팅 상태

```
체크리스트:
□ Google Analytics / Firebase Analytics 연동 여부
□ 광고 SDK (AdMob 등) 대시보드 접근 가능 여부
□ 인앱결제/구독 SDK (RevenueCat 등) 연동 여부
□ DAU/MAU 등 기본 지표 확인 가능 여부
□ 수익 데이터 (광고 수익, 결제 수익) 확인 가능 여부
```

**Output:** 측정 준비도 등급 & 맞춤 가이드

```yaml
measurement_readiness:
  grade: "A" | "B" | "C"

  # Grade C: 미설정 — 측정 인프라 자체가 없음
  # → 대시보드 제공 내용:
  #   1차 가치: "뭘 달아야 하는가" (GA/Firebase 세팅 가이드, AdMob 연동 가이드)
  #   2차 가치: "2주간 뭘 봐야 하는가" (핵심 지표 정의 + 일일 체크 루틴)
  #   → 2주 후 데이터 쌓이면 본 진단(Module A~E) 자동 활성화

  # Grade B: 부분 설정 — 일부 지표만 확인 가능
  # → 대시보드 제공 내용:
  #   보완 포인트 구체 안내 (예: "GA는 있는데 수익 추적이 없음 → AdMob 리포트 API 연결 필요")
  #   확인 가능한 데이터로 부분 진단 먼저 제공
  #   보완 완료 시 전체 진단 업데이트

  # Grade A: 완전 설정 — 바로 진단 가능
  # → Module A~E 즉시 실행 → 그로스 전략 바로 제안

  missing_items:
    - item: "firebase_analytics"
      guide: "Firebase Analytics 5분 세팅 가이드"
      priority: "필수"
    - item: "admob_reporting"
      guide: "AdMob 수익 리포트 연동 방법"
      priority: "필수"
    - item: "revenuecat"
      guide: "RevenueCat 세팅 (인앱결제 있는 경우만)"
      priority: "선택"
```

**프롬프트 지시사항:**
- Grade 판정 후 해당 Grade에 맞는 대시보드 콘텐츠를 활성화
- Grade C 고객에게 "지표가 없어서 진단 불가"라고 거절하지 말 것 — "지표 세팅이 첫 번째 처방"이라는 프레이밍
- 각 세팅 가이드는 "5분 안에 끝나는" 단위로 쪼개서 제공
- 2주 타임라인은 토스 CC 기준: 세팅(Day 1~3) → 데이터 축적(Day 4~10) → 유의미한 패턴 확인(Day 11~14)

#### Module A: 앱 프로필 파싱 & 정규화

**Input:** 고객이 폼에 입력한 Raw 데이터

```
앱 이름: 실검위젯
플랫폼: Android
카테고리: 유틸리티
DAU: 2500
MAU: 12000
수익 모델: AdMob (배너+전면)
월 수익: 40만원
운영 앱 수: 2개
목표 월 수익: 150만원
가장 큰 고민: 인앱결제를 달고 싶은데 뭘 유료로 만들어야 할지 모르겠어요
```

**Output:** 구조화된 앱 프로필

```yaml
app_profile:
  name: "실검위젯"
  platform: "Android"
  category: "utility"
  category_sub: "widget"
  
  metrics:
    dau: 2500
    mau: 12000
    stickiness: 0.208  # DAU/MAU — 습관적 사용도 지표
    estimated_sessions_per_day: 3.5  # 카테고리 평균 추정
    
  revenue:
    current_model: ["admob_banner", "admob_interstitial"]
    monthly_revenue_krw: 400000
    estimated_ecpm: 1600  # 역산: 40만원 / (2500 × 30 × 3.5 노출 / 1000)
    arpu_monthly: 33  # 40만원 / MAU 12000
    
  goals:
    target_monthly_revenue: 1500000
    revenue_gap: 1100000  # 목표 - 현재
    timeline: "not_specified"
    
  context:
    app_count: 2
    primary_concern: "monetization_model_uncertainty"
    attempted_solutions: []
```

**프롬프트 지시사항:**
- DAU/MAU 비율(Stickiness)을 반드시 계산. 이 수치가 수익 모델 추천의 핵심 변수
- 입력값이 비어있는 경우 카테고리 평균으로 추정하되, "추정값" 표시
- eCPM을 역산해서 현재 광고 효율을 진단
- ARPU(유저당 월 수익)를 산출해서 개선 여지 판단

#### Module B: 수익화 단계 진단

고객의 앱을 5단계 중 하나로 분류한다. 각 단계마다 다른 처방이 필요하기 때문.

**진단 기준표:**

| Level | 이름 | 조건 | 핵심 과제 |
|---|---|---|---|
| 1 | 수익 모델 없음 | 수익 모델 미적용, 또는 월 수익 0원 | 첫 수익 모델 선택 |
| 2 | 광고 단일 의존 | AdMob only, 인앱/구독 시도 없음 | 수익 모델 다변화 |
| 3 | 복합 수익 초기 | 광고 + 인앱/구독 시도 중, 전환율 < 카테고리 평균 | 전환율 최적화 |
| 4 | 수익 구조 안정 | 복합 수익, 전환율 ≥ 평균, 월 수익 100만원+ | 객단가/리텐션 최적화 |
| 5 | 스케일 준비 | 월 수익 300만원+, 유닛 이코노믹스 건전 | 성장 채널 확장 |

**분류 로직 (의사결정 트리):**

```
Q1: 수익 모델이 있는가?
  → No → Level 1
  → Yes → Q2

Q2: 광고 외 수익 모델(인앱결제/구독)이 있는가?
  → No → Level 2
  → Yes → Q3

Q3: 인앱/구독 전환율이 카테고리 평균 이상인가?
  → No → Level 3
  → Yes → Q4

Q4: 월 수익이 100만원 이상인가?
  → No → Level 3 (전환율은 괜찮지만 볼륨 부족)
  → Yes → Q5

Q5: 월 수익이 300만원 이상이고, CAC < LTV인가?
  → No → Level 4
  → Yes → Level 5
```

**프롬프트 지시사항:**
- 반드시 분류 근거를 명시 ("AdMob만 사용 중이고 인앱결제 시도 이력 없음 → Level 2")
- Level별 "전형적 증상"과 고객의 실제 상태를 매칭해서 보여주기
- "가장 큰 고민" 텍스트를 분석해서 감정 상태(불안, 정체감, 도피 패턴) 파악

#### Module C: 벤치마크 비교 분석

고객 앱의 지표를 동일 카테고리 벤치마크와 비교한다. "당신 앱은 지금 어디에 있는가"를 객관적으로 보여주는 역할.

**비교 항목:**

```
1. eCPM 비교
   → 고객 앱 eCPM vs 카테고리 평균 eCPM
   → 판정: 하위(개선 여지 큼) / 평균 / 상위(광고 최적화 완료)

2. ARPU 비교
   → 고객 앱 ARPU vs 카테고리 평균 ARPU
   → 특히 "광고 only ARPU" vs "광고+구독 ARPU" 갭을 보여줌

3. Stickiness(DAU/MAU) 비교
   → 고객 앱 vs 카테고리 평균
   → Stickiness가 높으면 구독 모델 적합, 낮으면 일회성 결제 적합

4. 구독/인앱 전환율 벤치마크
   → 카테고리별 평균 전환율 범위 제시
   → "당신 앱에서 전환율 2%는 현실적입니다" 근거

5. 수익 잠재력 갭
   → 현재 수익 vs 카테고리 상위 25% 수준 예상 수익
   → "같은 MAU에서 상위 앱들은 월 X만원을 벌고 있습니다"
```

**프롬프트 지시사항:**
- 벤치마크 데이터 시트(자산 #2)를 참조하여 비교
- 모든 비교에 "출처/근거"를 명시 (예: "유틸리티 앱 평균 eCPM $1.5~3.0, 출처: AppLovin 2024 리포트")
- "하위/평균/상위" 3단계로 시각적 판정
- 부족한 영역을 처방 우선순위와 연결

---

#### Module D: 수익 시뮬레이션 & 최적 모델 매칭

가장 핵심적인 모듈. "뭘 얼마에 팔면 얼마가 나온다"를 구체적 숫자로 보여준다.

**D-1: 최적 수익 모델 매칭**

앱 특성에 따라 적합한 수익 모델을 1~2개 추천한다.

매칭 로직:

```
IF Stickiness ≥ 0.20 (높은 일 사용 빈도):
  → 구독 모델 우선 추천
  → 근거: 습관적 사용 = 높은 구독 유지율 기대

IF Stickiness < 0.15 (낮은 사용 빈도) AND 가치 밀도 높음:
  → 일회성 인앱결제 추천
  → 근거: 사용 빈도 낮으면 월 구독 이탈 높음

IF 카테고리 = 게임:
  → 보상형 광고 + 인앱 아이템
  → 근거: 게임 경제 내 자연스러운 결제 유도

IF 카테고리 = 콘텐츠/정보:
  → 프리미엄 콘텐츠 잠금 (paywall)
  → 근거: 콘텐츠 자체가 가치, 무료 미리보기로 전환 유도

모든 경우:
  → 기존 광고 수익 유지하면서 추가 수익 모델 적용
  → "광고 제거"는 구독 번들의 일부로만 제안 (단독 X)
```

**D-2: 수익 시뮬레이션**

추천 모델별 예상 수익을 3개 시나리오로 제시한다.

```
[구독 모델 시뮬레이션]

Input:
  - MAU: 12,000
  - 추천 구독 가격: ₩2,900/월
  - 카테고리 평균 전환율 범위: 1.5% ~ 5%

시나리오:
  보수적 (하위 25%): MAU × 1.5% × ₩2,900 = 12,000 × 0.015 × 2,900 = ₩522,000/월
  기본 (중앙값):     MAU × 3.0% × ₩2,900 = 12,000 × 0.030 × 2,900 = ₩1,044,000/월
  낙관적 (상위 25%): MAU × 5.0% × ₩2,900 = 12,000 × 0.050 × 2,900 = ₩1,740,000/월

기존 광고 수익 유지: ₩400,000/월

합산 예상:
  보수적: ₩922,000/월
  기본:   ₩1,444,000/월
  낙관적: ₩2,140,000/월

목표(₩1,500,000) 달성 가능성: 기본 시나리오에서 근접, 낙관에서 초과
```

**D-3: 가격 추천 로직**

```
Step 1: 카테고리별 WTP(Willingness To Pay) 범위 참조
  → 유틸리티 앱 한국 시장: ₩1,900 ~ ₩4,900/월

Step 2: 가치 번들 설계
  → 단순 "광고 제거"만으로는 WTP 낮음
  → "광고 제거 + 프리미엄 기능 2~3개" 번들이 전환율 2~3배

Step 3: 가격 A/B 테스트 설계
  → 테스트 가격 A: ₩2,900/월
  → 테스트 가격 B: ₩3,900/월
  → 측정: 전환율 × 가격 = 총 수익 극대화 포인트
```

**프롬프트 지시사항:**
- 모든 시뮬레이션에 **계산식을 투명하게 공개** (가장 중요한 원칙)
- 전환율 가정의 근거를 반드시 명시 ("유틸리티 앱 평균 구독 전환율 1.5~5%, 출처: RevenueCat 2024 State of Subscription Apps")
- "이 숫자는 보장이 아닌 추정"임을 명시하되, 비관적으로 치우치지 말 것
- 가격 추천 시 반드시 A/B 테스트 설계를 포함 (처방의 신뢰도를 높이는 장치)

## 3. 웹 대시보드

### 3-1. 목적

최종 산출물은 **웹 대시보드**. 고객이 로그인하면 자신의 앱 진단 결과를 대시보드에서 탐색한다. 진단 과정, 분석 근거, 처방이 단계적으로 펼쳐지며, 언제든 돌아와서 확인할 수 있다.

PDF 대비 장점:
- 진단 과정을 **인터랙티브하게** 보여줄 수 있음 (펼치기/접기, 탭 전환)
- 측정 준비도에 따라 **콘텐츠가 동적으로 바뀜** (Grade C는 세팅 가이드, Grade A는 풀 진단)
- 2주 사용 기간 동안 **데이터가 쌓이면 대시보드가 업데이트**됨
- 구독 전환 시 연속 코칭 내용이 같은 대시보드에 누적

### 3-2. 대시보드 구조

측정 준비도(Grade)에 따라 보이는 섹션이 다르다.

```
═══════════════════════════════════════
  💊 앱 수익 처방전 — 대시보드
═══════════════════════════════════════

[Section 0] 측정 준비도 & 세팅 가이드    ← Grade B, C만 표시
[Section 1] 앱 프로필 요약               ← 전체
[Section 2] 수익화 단계 진단             ← Grade A, B (부분)
[Section 3] 벤치마크 비교               ← Grade A
[Section 4] 수익 시뮬레이션             ← Grade A
[Section 5] 처방 (실행 계획)            ← 전체 (Grade별 내용 다름)
[Section 6] 다음 스텝 & 구독 안내       ← 전체
```

**Grade별 대시보드 경험:**

| Grade | 첫 접속 시 | 2주 후 |
|---|---|---|
| C (미설정) | Section 0 + 1 + 5(세팅 로드맵) + 6 | 데이터 확인 후 Section 2~4 잠금 해제 |
| B (부분) | Section 0(보완) + 1 + 2(부분) + 5 + 6 | 보완 완료 시 전체 진단 활성화 |
| A (완전) | Section 1~6 전체 즉시 활성화 | 실행 결과 반영 업데이트 |

### 3-3. 섹션별 상세

#### Section 0: 측정 준비도 & 세팅 가이드 (Grade B, C만 표시)

```
┌─────────────────────────────────────────────┐
│ 🔧 측정 준비도 체크                          │
│    Grade: C — 측정 인프라 세팅 필요           │
├─────────────────────────────────────────────┤
│                                             │
```

## 4. 개발 환경 통합 (.apprx.md + CLI)

### 4-1. 목적

개발자가 **한 번 설치하면, 이후 결제 기간 동안 아무것도 안 해도** AI 코딩 어시스턴트가 수익화 맥락을 자동으로 참조한다. 웹 대시보드를 열지 않아도, 코딩하면서 자연스럽게 수익화 전략과 함께 일하게 된다.

### 4-2. 동작 원리

```
$ npx @apprx/cli init

이 한 줄이 하는 일:
1. API 키 인증 (대시보드에서 발급받은 키 입력)
2. 프로젝트 루트에 .apprx.md 파일 생성
3. AI 도구 자동 감지 & 연결:
   - Claude Code 감지 → CLAUDE.md에 .apprx.md 참조 추가
   - Cursor 감지 → .cursor/rules에 참조 추가
   - 기타 → .apprx.md 자체가 범용 컨텍스트로 작동
4. 완료. 이후 설정 없음.
```

### 4-3. .apprx.md 파일 구조

AI 어시스턴트가 읽기 최적화된 마크다운. 사람도 읽을 수 있지만, 주 독자는 AI다.

```markdown
# App Revenue Prescription — 실검위젯
# 이 파일은 앱 수익 처방전 서비스의 컨텍스트 파일입니다.
# AI 어시스턴트가 이 파일을 참조하여 수익화 전략에 맞는 코드와 조언을 제공합니다.

- 측정 준비도: Grade B (AdMob 있음, GA 미연동)
```

### 4-4. 자동 동기화

.apprx.md는 대시보드의 데이터와 동기화된다.

| 시점 | 동기화 방법 |
|---|---|
| 최초 설치 | `npx @apprx/cli init` |
| 이후 자동 | AI 도구가 프로젝트 열 때 백그라운드 동기화 (MCP 지원 시) |
| 수동 갱신 | `npx @apprx/cli sync` |
| 대시보드에서 | "컨텍스트 파일 다운로드" 버튼 → 복사/붙여넣기 |

**Phase 1(데모)에서는** 동기화 없이 정적 파일. Phase 3에서 API 연동.

### Phase 2: 홍보 & 사전 신청 수집 (목표: 1~2주)

데모 영상을 배포하고, 사전 신청을 수집한다.

| 작업 | 채널 | 목표 지표 |

| 1:1 DM 아웃리치 | 트위터/X, 인스타 인디 개발자 | 회신율 |

**사전 신청 폼 수집 항목:**
```
- 이름/닉네임
- 앱 이름 (선택)
- 카테고리
- 현재 월 수익 범위 (선택지)
- 가장 큰 고민 (자유 입력)
- 이메일
```

**Go/No-Go 기준:**
- Go: 사전 신청 10명 이상 → Phase 3 진입
- Pivot: 사전 신청 3~9명 → 영상/메시지 수정 후 재시도
- Stop: 사전 신청 2명 이하 + 2주 경과 → 가치 제안 재설계
